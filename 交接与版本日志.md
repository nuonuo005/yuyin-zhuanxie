# 语言转写：交接与版本日志

本文档写给后续接手本工程的开发者或 AI 编程工具。目标是避免“越修越乱”，在修改前先理解边界、备份、验证方式和回退办法。

## 1. 软件定位

软件名称：语言转写

当前定位：Windows 本地桌面语音输入工具。

核心流程：

```txt
热键录音 -> 本地 FunASR/Paraformer 中文 ASR -> 可选 DeepSeek 书面化 -> 写入剪贴板 -> 自动粘贴到用户当前输入位置
```

开发来源：

- 原始需求由用户提出。
- 当前版本由 Codex 在用户本机 `E:\AI linshi\语言转写` 中实现。
- 项目代码计划开源；应用代码可按 `LICENSE` 使用。
- 本项目不复制 VocoType 私有代码，只复用用户本机已经下载的公开 FunASR / ModelScope ONNX 模型目录。

重要边界：

- 不要提交或分发 VocoType 程序文件。
- 不要提交或分发 `.local_models/` 里的模型文件，除非确认模型协议允许并保留来源说明。
- 不要提交 `config.json`，里面可能有用户 API Key。
- 不要提交 `history.jsonl`，里面可能有用户语音转写隐私内容。

## 7. 常见问题与修理方式

### 7.1 双击后没有窗口

先运行：

```powershell
cd "E:\AI linshi\语言转写"
.\语言转写.bat doctor
```

再用命令行启动 GUI 查看错误：

```powershell
.\.venv\Scripts\python.exe -m yuyin_zhuanxie
```

常见原因：

- `.venv` 不存在。
- 依赖没装完整。
- `customtkinter` 缺失。
- Python 的 Tk/Tcl 环境损坏。

修理：

```powershell
.\install.ps1
```

### 7.2 悬浮窗不出现

检查配置：

```json
"show_floating_indicator": true
```

检查 `gui_modern.py`：

- `ensure_floating_button`
- `create_floating_button`
- `update_floating_button`

不要只在录音时创建悬浮窗，当前设计是程序运行即常驻。

### 7.3 转写慢

不要把 `asr.py` 改回每次重新加载模型。

当前关键函数：

```txt
get_engines()
warmup_models()
transcribe_audio()
```

正确设计：

```txt
客户端启动 -> warmup_models -> 模型常驻内存 -> 按热键时复用模型
```

### 7.4 没有粘贴到目标输入框

重点检查：

```txt
winfocus.py
gui_modern.py 中的 target_hwnd / last_external_hwnd / send_paste
```

当前设计：

1. 追踪最近外部窗口。
2. 开始录音时确定目标窗口。
3. 输出写入剪贴板。
4. 切回目标窗口。
5. 点击当前鼠标位置。
6. 发送 `Ctrl+V`。

如果某些软件禁止模拟粘贴，可能需要管理员权限运行本程序，或者改用更底层的 Windows SendInput。

### 7.5 DeepSeek 没有调用

检查：

```json
"enable_ai_polish": true
```

检查：

```json
"ai_provider": "deepseek"
```

检查 provider 里：

```json
"base_url": "https://api.deepseek.com/v1"
"model": "deepseek-chat"
"api_key": "不要打印出来"
```

如果关闭 `enable_ai_polish`，这是预期行为：只本地转写，不请求 DeepSeek。

### 7.6 热键无效

检查：

```txt
gui_modern.py -> register_hotkey()
```

依赖：

```txt
keyboard
```

某些管理员权限窗口可能要求本程序也以管理员权限运行。不要随便更换热键库，除非确认能继续支持全局热键。

## 8. 回退办法

如果被其他软件修坏：

### 8.1 有备份目录

关闭所有 `语言转写` 窗口和 `pythonw.exe` 进程，然后把备份复制回来。

建议先结束进程：

```powershell
Get-Process pythonw -ErrorAction SilentlyContinue | Stop-Process
```

再恢复备份。

### 8.2 有 Git

查看改动：

```powershell
git status
git diff
```

回到基线：

```powershell
git restore .
```

如果确认要完全回到某个提交：

```powershell
git reset --hard <commit_id>
```

注意：`git reset --hard` 会丢弃未保存改动，必须确认后再用。

### 8.3 没有备份也没有 Git

不要继续让 AI 乱改。先复制当前坏掉的目录留档，然后按 `README.md` 和本文档重新修：

```powershell
Copy-Item "E:\AI linshi\语言转写" "E:\AI linshi\语言转写_broken_snapshot" -Recurse
```

再从最后一个可用版本重建。

## 12. 当前未解决问题：内容区底部留白

### 现象

用户在「部署与开源」「录音与输出」等页面截图，发现白色内容区下方仍有一块灰色空白，无法贴到窗口底部。

### 已尝试的修复（均未彻底生效）

1. 把 `content` 框架的 `pady` 从 `(4, 24)` 逐步改为 `(4, 12)`、`4`。
2. 把「部署与开源」页的 `CTkScrollableFrame` 换成 `tk.Canvas + CTkScrollbar` 手动滚动。
3. 把「部署与开源」页换成 `CTkTextbox`（原 `tk.Text` 子类，滚动最稳定）。
4. 在「录音与输出」「快捷键绑定」页外加一层白色卡片，并设置 `grid_rowconfigure(0, weight=1)` 让其垂直撑满。

### 问题根因分析

CustomTkinter 的 `CTkScrollableFrame` 和 `CTkTextbox` 内部都依赖 tkinter 的 `Canvas` 或 `Text` 组件。虽然外层 `self.content` 已经设置了 `grid_rowconfigure(0, weight=1)`，子控件也设置了 `sticky="nsew"`，但这些 CustomTkinter 封装组件在以下场景下不会真正填满父容器：

- **高 DPI / 缩放**：Windows 显示缩放非 100% 时，CustomTkinter 内部 Canvas 的 `scrollregion` 和 `reqheight` 计算容易偏小，导致内容高度没有撑满可用空间。
- **最小高度限制**：`CTkTextbox` 内部 `Text` 组件有默认最小高度；即使父容器给了更多空间，它也可能不扩展，具体取决于 CustomTkinter 版本和 DPI。
- **grid 权重传播**：`CTkScrollableFrame` 的外层 Frame 会按内部内容自适应，不会自动吸收父容器剩余空间，除非显式 `grid_propagate(False)` 并绑定高度。
- **时机问题**：在页面渲染完成前读取 `self.content.winfo_height()` 可能得到旧值，导致设置的高度小于实际可用高度。

### 建议修复方向

后续接手者建议从以下方向入手，按推荐度排序：

1. **显式绑定高度（最稳）**
   - 在 `show_about()`、`show_recording()`、`show_hotkeys()` 等页面渲染完成后，用 `self.after()` 延迟读取 `self.content.winfo_height()`。
   - 把 `CTkTextbox` / `CTkScrollableFrame` 的 `height` 直接设为这个值（减去内边距）。
   - 同时监听窗口 `Configure` 事件，在窗口大小变化时重新设置高度。

2. **使用普通 tkinter 组件替换 CustomTkinter 滚动组件**
   - `tk.Text` + `tk.Scrollbar` 比 `CTkTextbox` 更可控，可以明确设置 `height` 行数或像素。
   - `tk.Canvas` + `tk.Scrollbar` 手动实现滚动，避免 `CTkScrollableFrame` 的内部高度计算 bug。

3. **在内容区底部添加一个可伸展的填充器**
   - 如果白卡片无法完全撑满，可以在其下方加一个同背景色的 `CTkFrame`，设置 `grid_rowconfigure(..., weight=1)` 让它吸收剩余空间。
   - 这样视觉上没有断层，但治标不治本。

4. **降级 CustomTkinter 或换 UI 框架**
   - 如果 CustomTkinter 在该 Windows 版本/DPI 下持续出现布局异常，可考虑换 `ttkbootstrap`、`PyQt`、`PySide` 或纯 `tkinter`。
   - 工程量大，仅作为最后手段。
